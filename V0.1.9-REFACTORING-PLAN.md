# v0.1.9 Architecture Simplification Plan

**Goal:** Reduce CLI module count from ~30 → ~15 while preserving modularity, testability, and supporting future feature growth.

**Strategy:** Group by layer (formatters, services, commander) — merge small files, keep commands separate.

---

## Target Structure

```
src/ghostclaw/
├── cli/
│   ├── ghostclaw.py         # Main entry (unchanged)
│   ├── commander.py         # Merged: base + registry
│   ├── services.py          # Merged: analyzer_service, pr_service, config_service
│   ├── formatters.py        # Merged: base, json, markdown, terminal
│   └── commands/
│       ├── analyze.py       # Updated imports
│       ├── debug.py         # Already added
│       ├── bridge.py
│       ├── doctor.py
│       ├── init.py
│       ├── test.py
│       └── plugins/         # Keep separate (8 files)
└── core/                    # No changes (domain layer)
```

**Reduction:** `cli/` goes from 20+ files → 10 files.

## Risk Mitigation

- **Each phase is reversible** (Git history preserves original files)
- **Full test suite passes** after each merge (150+ tests)
- **No logic changes** — pure file reorganization
- **Static analysis**: `python -m py_compile` on entire tree to catch missing imports
- **Pre-commit hooks**: ensure imports are updated

---

## Benefits for Long-Term Scalability

1. **Fewer files to navigate** → faster onboarding
2. **Clearer layer boundaries** → formatters/services/commander are now explicit groups
3. **Easier to find code** → "Where is X?" answer: either `formatters.py`, `services.py`, `commander.py`, or `commands/`
4. **Reduced cognitive load** — developers don't need to jump between 5 formatter files
5. **Future feature add** — new formatter? Just add a class to `formatters.py`. New service? Add to `services.py`.
6. **No circular dependency risk** if we maintain dependency direction: commands → services → core

## Alternatives Considered (and Rejected)

- **Keep as-is**: Too many files, confusing for newcomers
- **Merge commands too**: Would blur user-facing feature boundaries; commands should stay separate
- **Group by feature** (e.g., all analyze-related together): Would mix layers; not recommended

Our chosen approach (layer grouping) is standard in mature Python projects (e.g., Django's `django/forms` used to be many files, now `forms/__init__.py` re-exports many but core logic sits in consolidated modules).
